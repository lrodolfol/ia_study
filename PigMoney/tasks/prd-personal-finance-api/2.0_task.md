# Task 2.0: Repository Layer & Database Setup

<critical>Read the prd-personal-finance-api.md and techspec-personal-finance-api.md files in this folder; if you do not read these files, your task will be invalidated</critical>

## Overview

Implement the data access layer using Entity Framework Core and PostgreSQL. Create the AppDbContext with entity configurations using Fluent API, implement the generic IRepository<T> interface, create 6 repository implementations (one per aggregate root), generate the initial migration with database schema including indexes and triggers, and configure connection pooling for optimal performance.

<requirements>
- Use Entity Framework Core with Npgsql provider for PostgreSQL
- Implement IRepository<T> interface in Domain layer (abstraction)
- Create AppDbContext with entity configurations using Fluent API
- One repository implementation per aggregate root (6 repositories)
- Global query filter for soft deletes: IsDeleted == false
- Database trigger for expense items sum validation
- Indexes on date, category_id, account_id columns for performance
- Connection pooling: Min=5, Max=100 connections
- Lazy loading disabled (explicit includes only)
- Connection string from appsettings.json (no sensitive data in code)
- Follow Repository RULES.md standards
</requirements>

## Implementation Details

Reference techspec-personal-finance-api.md sections:
- **Database Schema** (lines 327-428): Complete SQL schema with tables, indexes, and trigger
- **System Architecture - Repository Layer** (lines 21-27): Repository layer structure and responsibilities
- **Integration Points - PostgreSQL** (lines 476-488): Connection configuration and EF Core setup
- **Main Interfaces - IRepository<T>** (lines 71-79): Repository interface definition

Key implementation points:
- Entity configurations use Fluent API (not data annotations)
- Decimal properties use HasPrecision(18, 2)
- Foreign key relationships configured with OnDelete behavior
- Global query filter: modelBuilder.Entity<T>().HasQueryFilter(e => !e.IsDeleted)
- Trigger function and trigger for expense_items sum validation in migration
- Indexes with WHERE is_deleted = FALSE for performance
- Repository pattern returns Result<T> for all operations
