### MAJOR Issues

**1. UpdateAsync no controller retorna 404 para erro de "Category not found" (BudgetRoutes.cs, linha 55)**

No metodo `UpdateAsync` do `BudgetRoutes.cs`, qualquer falha do servico retorna `NotFound (404)`. Porem, o `BudgetService.UpdateAsync` pode falhar por dois motivos distintos: (a) Budget nao encontrado e (b) Category nao encontrada. Quando o Budget existe mas a Category referenciada e invalida, a resposta correta seria `400 BadRequest`, nao `404 NotFound`.

Este e um padrao pre-existente no projeto (o `IncomeRoutes` tem o mesmo problema), mas como a task 5.0 criou este codigo, vale registrar. O controller nao diferencia os tipos de erro retornados pelo servico.

**Sugestao de correcao:**
```csharp
[HttpPut("{id}")]
public async Task<IActionResult> UpdateAsync(int id, [FromBody] UpdateBudgetRequest request)
{
    var result = await _budgetService.UpdateAsync(id, request);
    if (!result.IsSuccess)
    {
        if (result.Error.Contains("not found", StringComparison.OrdinalIgnoreCase)
            && result.Error.Contains("Budget", StringComparison.OrdinalIgnoreCase))
            return NotFound(new { statusCode = 404, message = result.Error, error = Array.Empty<string>() });

        return BadRequest(new { statusCode = 400, message = result.Error, error = Array.Empty<string>() });
    }
    return Ok(result.Value);
}
```

Alternativamente, evoluir o `Result<T>` para incluir um enum de tipo de erro (NotFound, ValidationError, etc.) seria uma solucao mais robusta e desacoplada de strings.

### MINOR Issues

**1. Testes unitarios do BudgetService nao seguem a convencao de nomes com "should" (BudgetServiceTests.cs)**

A convencao do projeto (rules de testes) indica que nomes de testes devem comecar com "should". Os testes atuais usam o padrao `MetodoAsync_Condicao_ShouldResultado`, o que e aceitavel e legivel, mas nao comeca com "should". Exemplo: `CreateAsync_ShouldReturnSuccess` poderia ser `ShouldReturnSuccess_WhenCreatingWithValidData`.

Nota: Este padrao ja e utilizado nos validadores (`ShouldHaveError_WhenLimitAmountIsZero`), entao ha inconsistencia interna nos testes desta task.

**2. Duplicacao entre CreateBudgetRequestValidator e UpdateBudgetRequestValidator**

Os dois validadores sao identicos. Embora isso siga o padrao do projeto para outros recursos, e uma oportunidade de refatoracao futura -- poderia-se extrair um validador base `BudgetRequestValidatorBase` com as regras comuns.

**3. BudgetService.CreateAsync define CreatedAt/UpdatedAt manualmente (BudgetService.cs, linhas 49-50)**

O servico define `CreatedAt = DateTime.UtcNow` e `UpdatedAt = DateTime.UtcNow` manualmente na criacao da entidade. Se o EF Core ou o `AppDbContext` ja gerencia essas propriedades automaticamente (via `SaveChanges` override), essa logica seria redundante. Se nao gerencia, a abordagem esta correta, mas seria mais robusto centralizar isso no `AppDbContext`.

## Recommendations

1. **Diferenciar tipos de erro no controller UpdateAsync**: O `BudgetRoutes.UpdateAsync` deve distinguir entre "Budget not found" (404) e "Category not found" (400). Isso evita confusao para o consumidor da API. Considerar evoluir o `Result<T>` com um enum de tipo de erro para toda a aplicacao.

2. **Centralizar CreatedAt/UpdatedAt**: Se ainda nao implementado, considerar um override de `SaveChangesAsync` no `AppDbContext` para definir automaticamente `CreatedAt` e `UpdatedAt` em todas as entidades que herdam de `BaseEntity`, evitando duplicacao em cada servico.

3. **Padronizar nomes de testes**: Alinhar os nomes dos testes unitarios do `BudgetServiceTests` com o padrao "Should..." ja usado nos testes de validadores.

4. **Considerar extrair validador base**: Para reduzir duplicacao entre `CreateBudgetRequestValidator` e `UpdateBudgetRequestValidator`, considerar um validador base compartilhado. Isso e uma melhoria de baixa prioridade, valida apenas se mais campos forem adicionados no futuro.
