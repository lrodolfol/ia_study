# Task 2.0: Category & Account Resources

<critical>Read the prd.md and techspec.md files in this folder; if you do not read these files, your task will be invalidated</critical>

## Overview

Implement the full vertical slice for **Category** and **Account** — these two resources have no FK dependencies and must be delivered together as the foundation for all other resources. Deliverable includes entity, EF configuration, repository, service, controller, validators, DTOs, and EF migration.

<requirements>
- Category and Account entities inherit BaseEntity
- ICategoryRepository and IAccountRepository extend IRepository<T> (no extra filter methods needed)
- Generic Repository<T> base class in Repository layer implements IRepository<T>
- CategoryRepository and AccountRepository extend Repository<T>
- CategoryService and AccountService implement their interfaces using Result<T>
- FluentValidation validators for Create and Update requests for both resources
- MVC controllers with routes: GET list (paginated), GET by id, POST (201), PUT (200), DELETE (409 if dependents)
- EF IEntityTypeConfiguration<T> for both entities; one migration covering both
- All list responses use PaginatedList<T> defined in Application/DTOs/Common/PaginatedList.cs
- DELETE account: returns 409 if account has linked incomes or expenses
- DELETE category: returns 409 if category has linked expenses or budgets
- All .cs files must start with the required `//created by:` comment header
</requirements>

## Success Criteria

- GET `/api/v1/accounts?page=1&pageSize=10` returns `PaginatedList<AccountResponse>` wrapped in standard envelope
- GET `/api/v1/categories/{id}` returns 404 for unknown ID
- POST `/api/v1/accounts` returns 201 with created resource
- DELETE `/api/v1/categories/{id}` returns 409 when category has linked data
- FluentValidation returns 400 with error list on invalid request body
- `dotnet build` passes with zero errors
