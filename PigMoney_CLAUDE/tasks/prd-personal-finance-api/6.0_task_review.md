## Issues Found

### CRITICAL Issues

**1. Exposicao de mensagem de excecao em producao (Seguranca)**
- **Arquivo**: `src/API/Middleware/ExceptionHandlerMiddleware.cs`, linha 36
- **Descricao**: O middleware inclui `ex.Message` diretamente no campo `Error` da resposta 500. Em ambiente de producao, isso pode vazar detalhes internos da aplicacao (stack traces parciais, nomes de tabelas do banco de dados, connection strings em excecoes de conexao, etc.) para o cliente.
- **Sugestao**: Retornar uma mensagem generica para o cliente e logar a mensagem real apenas no servidor (o que ja e feito corretamente na linha 26). Opcionalmente, manter `ex.Message` apenas em ambiente Development.

```csharp
var errorMessage = context.RequestServices
    .GetService<IWebHostEnvironment>()?.IsDevelopment() == true
    ? ex.Message
    : "An internal error occurred. Please try again later.";

var response = new StandardApiResponse<object>(
    StatusCode: 500,
    Message: "An unexpected error occurred.",
    Error: new[] { errorMessage },
    Data: default
);
```

### MAJOR Issues

**1. Catch vazio no ResponseWrapperMiddleware silencia excecoes**
- **Arquivo**: `src/API/Middleware/ResponseWrapperMiddleware.cs`, linhas 78-81 e 93-96
- **Descricao**: Os blocos `catch` nos metodos `IsAlreadyWrapped` e `DeserializeBody` capturam todas as excecoes e retornam silenciosamente (`return false` / `return body`). A regra do projeto em CLAUDE.md diz "Never silence exceptions". Embora esses metodos sejam defensivos por natureza (parsing de JSON pode falhar), seria recomendavel ao menos logar em nivel DEBUG para facilitar a depuracao.
- **Sugestao**: Injetar `ILogger` no middleware ou, no minimo, documentar explicitamente por que esses catches sao intencionais. Uma alternativa seria usar `JsonDocument.TryParse` (se disponivel) ou um metodo auxiliar que nao lance excecao.

**2. StreamReader nao esta sendo disposed no ResponseWrapperMiddleware**
- **Arquivo**: `src/API/Middleware/ResponseWrapperMiddleware.cs`, linha 31
- **Descricao**: O `StreamReader` criado na linha 31 nao esta dentro de um bloco `using`. Embora o `MemoryStream` subjacente esteja com `using`, o `StreamReader` em si pode reter recursos temporarios.
- **Sugestao**: Usar `using` no StreamReader ou `await using`:

```csharp
using var reader = new StreamReader(memoryStream, leaveOpen: true);
var responseBody = await reader.ReadToEndAsync();
```

### MINOR Issues

**1. Duplicacao de JsonSerializerOptions entre middlewares**
- **Arquivos**: `src/API/Middleware/ExceptionHandlerMiddleware.cs` (linha 13) e `src/API/Middleware/ResponseWrapperMiddleware.cs` (linha 11)
- **Descricao**: Ambos os middlewares declaram instancias `static readonly` identicas de `JsonSerializerOptions` com `CamelCase` policy. Isso e uma duplicacao que poderia ser centralizada.
- **Sugestao**: Criar uma classe `JsonDefaults` em `API/Models` ou similar, com uma propriedade estatica compartilhada:

```csharp
public static class JsonDefaults
{
    public static readonly JsonSerializerOptions CamelCase = new()
    {
        PropertyNamingPolicy = JsonNamingPolicy.CamelCase
    };
}
```

**2. Metodo `ExtractErrors` com complexidade ciclomatica elevada**
- **Arquivo**: `src/API/Middleware/ResponseWrapperMiddleware.cs`, linhas 109-127
- **Descricao**: O metodo tem multiplos caminhos condicionais com diferentes tipos de verificacao (status code, tipo de JsonElement, propriedades). Embora funcional, poderia ser refatorado para melhor legibilidade.
- **Sugestao**: Considerar extrair os diferentes caminhos em metodos privados com nomes descritivos.

**3. Falta de teste para cenario de 409 (Conflict) no ResponseWrapperMiddleware**
- **Arquivo**: `src/pigMoney.Tests/API/Middleware/ResponseWrapperMiddlewareTests.cs`
- **Descricao**: O `GetMessageForStatusCode` mapeia 409 para "Conflict", mas nao existe teste unitario cobrindo este cenario especifico no middleware.
- **Sugestao**: Adicionar um teste `InvokeAsync_When409_ShouldWrapWithConflictMessage`.

**4. Resposta 204 (NoContent) sendo envolvida pelo middleware**
- **Arquivo**: `src/API/Middleware/ResponseWrapperMiddleware.cs`, linhas 43-56
- **Descricao**: Respostas 204 (NoContent) sao semanticamente "sem corpo". O middleware esta transformando essas respostas em um JSON com `data: null`, o que modifica a semantica HTTP. Alguns clientes HTTP podem nao esperar corpo em respostas 204. Os testes de integracao verificam apenas o status code 204, sem validar se o corpo realmente contem o envelope -- o que pode ocultar problemas futuros.
- **Sugestao**: Considerar pular o wrapping para respostas 204, ou ao menos documentar a decisao de design.

## Positive Highlights

- **Record type**: Uso correto de `record` para `StandardApiResponse<T>`, aproveitando a imutabilidade e value equality do C#.
- **Primary constructors**: Os middlewares e route controllers utilizam primary constructors conforme exigido pelas regras do projeto.
- **Middleware ordering**: O `ExceptionHandlerMiddleware` e registrado antes do `ResponseWrapperMiddleware` em `Program.cs` (linhas 38-39), garantindo que excecoes nao tratadas sejam capturadas antes do wrapping.
- **Swagger bypass**: O `ResponseWrapperMiddleware` verifica e ignora requisicoes Swagger (metodo `IsSwaggerRequest`), evitando quebrar a UI do Swagger.
- **Double-wrap prevention**: O metodo `IsAlreadyWrapped` verifica se a resposta ja esta no formato envelope, evitando dupla encapsulacao -- cenario que ocorreria quando o `ExceptionHandlerMiddleware` ja formatou a resposta 500.
- **Testes bem estruturados**: Os testes unitarios seguem o padrao AAA e cobrem cenarios relevantes (200, 201, 204, 400, 404, 500, Swagger bypass, double-wrap).
- **Testes de integracao atualizados**: Todos os 4 arquivos de teste de integracao foram corretamente atualizados com o metodo `ReadDataAsync<T>` para deserializar o campo `data` do envelope.
- **Header de autoria presente em todos os arquivos**: Todos os `.cs` criados/modificados contem o comentario `//created by:` exigido.
- **FluentValidation configurado corretamente**: A auto-validacao e o registro de validators por assembly estao corretos em `Program.cs`.
- **Route controllers limpos**: Os controllers retornam dados simples (`Ok(result.Value)`, `NotFound(result.Error)`, etc.), delegando a formatacao do envelope ao middleware.

## Recommendations

1. **[CRITICAL] Remover `ex.Message` da resposta de producao** no `ExceptionHandlerMiddleware`. Retornar mensagem generica para o cliente e manter o log detalhado apenas no servidor. Condicionar a mensagem detalhada ao ambiente Development.

2. **[MAJOR] Tratar os catches silenciosos** em `IsAlreadyWrapped` e `DeserializeBody`. Ao menos logar em nivel Debug ou usar abordagens que evitem excecoes (como `TryParse`).

3. **[MAJOR] Adicionar `using` ao StreamReader** na linha 31 do `ResponseWrapperMiddleware` para garantir disposicao adequada de recursos.

4. **[MINOR] Centralizar `JsonSerializerOptions`** para evitar duplicacao entre os dois middlewares.

5. **[MINOR] Avaliar o tratamento de respostas 204** -- considerar se faz sentido pular o wrapping para NoContent, mantendo a semantica HTTP correta.

6. **[MINOR] Adicionar teste para cenario 409** no `ResponseWrapperMiddlewareTests`.
