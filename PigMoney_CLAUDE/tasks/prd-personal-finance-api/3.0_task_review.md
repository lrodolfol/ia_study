## Issues Found

### CRITICOS

### MAJOR

**1. IIncomeRepository.GetFilteredAsync tem 5 parametros (viola regra de max 3 parametros)**
- **Arquivo**: `src/Domain/Interfaces/IIncomeRepository.cs`, linha 8
- **Regra violada**: rules.naming.md - "Avoid passing more than 3 parameters; prefer using objects when needed"
- **Descricao**: O metodo `GetFilteredAsync` recebe 5 parametros (`startDate`, `endDate`, `accountId`, `page`, `pageSize`). O metodo `CountFilteredAsync` recebe 3 parametros, o que esta dentro do limite.
- **Sugestao**: Utilizar o DTO `IncomeFilterParams` ja existente e adicionar paginacao a ele, ou criar um parameter object dedicado:

```csharp
public record IncomeFilterQuery(DateTime? StartDate, DateTime? EndDate, int? AccountId, int Page, int PageSize);

public interface IIncomeRepository : IRepository<Income>
{
    Task<IEnumerable<Income>> GetFilteredAsync(IncomeFilterQuery query);
    Task<int> CountFilteredAsync(IncomeFilterQuery query);
}
```

**2. Income entity usa public setters em todas as propriedades**
- **Arquivo**: `src/Domain/Entities/Income.cs`, linhas 8-12
- **Regra violada**: CLAUDE.md - "Class fields must be private or private readonly"
- **Descricao**: Todas as propriedades (`Amount`, `Date`, `Description`, `AccountId`, `Account`) possuem `{ get; set; }` publicos, permitindo mutacao irrestrita. Nota: este e um pattern herdado de `BaseEntity` e consistente com as outras entidades do projeto (Account, Category), entao e um problema arquitetural mais amplo, nao especifico desta task.
- **Impacto**: Baixo neste contexto, pois EF Core requer setters acessiveis. Porem, `init` accessors ou `private set` seriam mais seguros.

**3. IncomeRoutes.GetAllAsync nao verifica `result.IsSuccess` antes de acessar `result.Value`**
- **Arquivo**: `src/API/Routes/IncomeRoutes.cs`, linha 25
- **Descricao**: O metodo `GetAllAsync` retorna `Ok(result.Value)` sem verificar se o resultado foi bem-sucedido. Embora o `GetAllAsync` do service atualmente sempre retorne `Success`, isso cria uma fragilidade caso o service evolua para retornar `Failure` em algum cenario futuro.
- **Sugestao**:

```csharp
[HttpGet]
public async Task<IActionResult> GetAllAsync(...)
{
    var filters = new IncomeFilterParams(startDate, endDate, accountId);
    var result = await _incomeService.GetAllAsync(filters, page, pageSize);
    if (!result.IsSuccess)
        return BadRequest(new { statusCode = 400, message = result.Error, error = Array.Empty<string>() });
    return Ok(result.Value);
}
```

Nota: este mesmo pattern existe em `AccountRoutes.GetAllAsync`, indicando um problema sistematico.

### MINOR

**4. Duplicacao da logica de filtragem no IncomeRepository**
- **Arquivo**: `src/Repository/Repositories/IncomeRepository.cs`, linhas 11-45
- **Descricao**: Os metodos `GetFilteredAsync` e `CountFilteredAsync` duplicam a mesma logica de construcao de query (filtros por `startDate`, `endDate`, `accountId`). Se um novo filtro for adicionado, sera necessario alterar ambos os metodos.
- **Sugestao**: Extrair um metodo privado para construir a query filtrada:

```csharp
private IQueryable<Income> ApplyFilters(DateTime? startDate, DateTime? endDate, int? accountId)
{
    IQueryable<Income> query = DbSet.AsQueryable();
    if (startDate.HasValue)
        query = query.Where(i => i.Date >= startDate.Value);
    if (endDate.HasValue)
        query = query.Where(i => i.Date <= endDate.Value);
    if (accountId.HasValue)
        query = query.Where(i => i.AccountId == accountId.Value);
    return query;
}
```

**5. Income entity usa `null!` para a navigation property Account**
- **Arquivo**: `src/Domain/Entities/Income.cs`, linha 12
- **Descricao**: `public Account Account { get; set; } = null!;` e um pattern comum com EF Core para evitar warnings de nullability, mas semanticamente fere a regra de "Never return null". E aceitavel neste contexto do EF Core, porem vale registrar.

**6. IncomeRoutes.CreateAsync retorna erro 400 quando account nao existe**
- **Arquivo**: `src/API/Routes/IncomeRoutes.cs`, linha 42
- **Descricao**: Quando a conta nao e encontrada, o service retorna `Failure("Account not found.")` e a rota retorna 400 (Bad Request). Semanticamente, 422 (Unprocessable Entity) seria mais apropriado para indicar que a entidade referenciada nao existe, diferenciando de um erro de validacao de formato.

**7. Testes unitarios do IncomeService nao testam cenario de UpdateAsync com account inexistente**
- **Arquivo**: `src/pigMoney.Tests/Application/Services/IncomeServiceTests.cs`
- **Descricao**: O metodo `UpdateAsync` possui validacao de existencia da conta (linha 65-66 do service), mas nao ha teste cobrindo o cenario onde o income existe mas a account nao. Apenas testa income nao encontrado e update bem-sucedido.
- **Sugestao**: Adicionar teste:

```csharp
[Fact]
public async Task UpdateAsync_WhenAccountNotFound_ShouldReturnFailure()
{
    var income = new Income { Id = 1, Amount = 1000m, Date = DateTime.UtcNow, Description = "Test", AccountId = 1 };
    _incomeRepositoryMock.Setup(r => r.GetByIdAsync(1)).ReturnsAsync(income);
    _accountRepositoryMock.Setup(r => r.ExistsAsync(99)).ReturnsAsync(false);

    var result = await _service.UpdateAsync(1, new UpdateIncomeRequest(2000m, DateTime.UtcNow, "Updated", 99));

    Assert.False(result.IsSuccess);
    Assert.Equal("Account not found.", result.Error);
}
```

## Recommendations

1. **Refatorar parametros do repositorio**: Criar um parameter object para `GetFilteredAsync` e `CountFilteredAsync` para respeitar a regra de max 3 parametros. Isso tambem elimina a duplicacao de assinatura.

2. **Extrair metodo de filtragem no repositorio**: Criar metodo privado `ApplyFilters` para eliminar duplicacao de logica de query entre `GetFilteredAsync` e `CountFilteredAsync`.

3. **Adicionar verificacao de `IsSuccess` no GetAllAsync da rota**: Mesmo que atualmente o service sempre retorne sucesso, a verificacao previne bugs futuros. Aplicar tambem em AccountRoutes.GetAllAsync para consistencia.

4. **Adicionar teste de UpdateAsync com account inexistente**: Garantir cobertura completa dos branches de validacao do service.

5. **Considerar HTTP 422 para referencia invalida**: Diferenciar erro de validacao de formato (400) de referencia a entidade inexistente (422) na rota de criacao.
