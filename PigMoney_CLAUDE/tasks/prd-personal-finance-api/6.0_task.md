# Task 6.0: Cross-Cutting Concerns

<critical>Read the prd.md and techspec.md files in this folder; if you do not read these files, your task will be invalidated</critical>

## Overview

Wire all cross-cutting concerns that apply globally to every API response: `ResponseWrapperMiddleware` (standard envelope), `ExceptionHandlerMiddleware` (500 fallback), `StandardApiResponse<T>` model, FluentValidation auto-registration, and Swagger/OpenAPI setup. After this task every endpoint returns the standardized `{ statusCode, message, error, data }` envelope.

<requirements>
- `StandardApiResponse<T>` record: `int StatusCode`, `string Message`, `IEnumerable<string> Error`, `T? Data`
- `ResponseWrapperMiddleware` wraps every successful response body in StandardApiResponse envelope
- `ExceptionHandlerMiddleware` catches unhandled exceptions and returns 500 StandardApiResponse with logged error
- FluentValidation errors are caught and returned as 400 with list of error messages in the `error` field
- Middleware registered in correct pipeline order in `Program.cs`
- Swagger/OpenAPI configured and accessible at `/swagger` in Development environment
- ILogger<T> used in ExceptionHandlerMiddleware (no Console.WriteLine)
- All .cs files must start with the required `//created by:` comment header
</requirements>

## Success Criteria

- Every GET/POST/PUT/DELETE response body matches: `{ "statusCode": int, "message": string, "error": [], "data": ... }`
- POST with invalid body returns 400 with `error` array populated by FluentValidation messages
- Triggering an unhandled exception returns 500 with `error` field and logs the exception at Error level
- GET unknown resource returns 404 with standard envelope
- Swagger UI loads at `/swagger` in Development
