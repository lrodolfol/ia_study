### Major Issues

**1. CreatedAtAction cross-controller pode nao funcionar em runtime**

- **Arquivo**: `src/API/Routes/ExpenseRoutes.cs`, linha 86
- **Descricao**: A chamada `CreatedAtAction("GetItemById", "ExpenseItemRoutes", new { id = result.Value!.Id }, result.Value)` referencia o controller `"ExpenseItemRoutes"`. O ASP.NET Core por convencao remove o sufixo `Controller` do nome da classe ao resolver controllers, mas nao remove o sufixo `Routes`. Como a classe se chama `ExpenseItemRoutes` e nao `ExpenseItemController`, o nome do controller para resolucao de rotas sera `"ExpenseItemRoutes"` (inteiro). Isso pode funcionar, mas e fragil e nao segue a convencao padrao do ASP.NET Core. Alem disso, o action name `"GetItemById"` precisa corresponder a `GetItemByIdAsync` (o ASP.NET Core remove o sufixo `Async` por padrao, resultando em `"GetItemById"` -- nesse caso especifico deve funcionar).
- **Sugestao**: Verificar manualmente que o `Location` header retornado no 201 aponta para a URL correta. Alternativamente, considerar usar `CreatedAtRoute` com rotas nomeadas para evitar ambiguidade.

**2. Ausencia de testes para UpdateExpenseRequestValidator e UpdateExpenseItemRequestValidator**

- **Arquivo**: Diretorio `src/pigMoney.Tests/Application/Validators/`
- **Descricao**: A task define explicitamente 4 validators (Create e Update para Expense e ExpenseItem), mas apenas os validators de Create possuem testes. Os validators `UpdateExpenseRequestValidator` e `UpdateExpenseItemRequestValidator` nao possuem arquivos de teste correspondentes. Embora as regras de validacao sejam identicas as dos Create validators, a boa pratica exige cobertura independente para garantir que alteracoes futuras nao introduzam regressoes.
- **Sugestao**: Criar `UpdateExpenseRequestValidatorTests.cs` e `UpdateExpenseItemRequestValidatorTests.cs` com os mesmos cenarios de teste dos Create validators.

### Minor Issues

**1. Inconsistencia semantica entre DeleteBehavior.Cascade e regra de negocio 409**

- **Arquivo**: `src/Repository/Configurations/ExpenseItemConfiguration.cs`, linha 47
- **Descricao**: A configuracao do banco define `OnDelete(DeleteBehavior.Cascade)` para a relacao ExpenseItem -> Expense, o que significa que deletar um Expense no banco cascatearia a delecao dos itens. Porem, o `ExpenseService.DeleteAsync` bloqueia a delecao com erro quando existem itens (retornando 409). Embora funcione como defense-in-depth, a semantica e contraditoria: a aplicacao diz "voce nao pode deletar" enquanto o banco diz "eu deletaria tudo junto". Se um dia alguem contornar a camada de servico, os itens seriam deletados silenciosamente.
- **Sugestao**: Considerar usar `DeleteBehavior.Restrict` ao inves de `Cascade`, alinhando banco e aplicacao na mesma semantica de protecao. Isso faria o banco rejeitar a delecao tambem.

**2. Comparacao de string magica no controller para distinguir 409 de 404**

- **Arquivo**: `src/API/Routes/ExpenseRoutes.cs`, linhas 70-71
- **Descricao**: O controller usa `string.Equals(result.Error, "Cannot delete expense with existing items.", StringComparison.Ordinal)` para decidir entre 409 e 404. Essa logica depende de uma string magica que esta duplicada entre o service e o controller. Qualquer alteracao na mensagem de erro do service quebraria a logica de status code do controller.
- **Sugestao**: Considerar enriquecer o `Result<T>` com um campo de status/codigo de erro (ex: um enum `ErrorType { NotFound, Conflict, Validation }`), ou criar constantes compartilhadas para as mensagens de erro. Isso tambem beneficiaria outros controllers que possam precisar da mesma logica.

**3. Parametros no record CreateExpenseRequest excedem o limite de 3**

- **Arquivo**: `src/Application/DTOs/Expenses/CreateExpenseRequest.cs`, linha 4
- **Descricao**: O record `CreateExpenseRequest` possui 5 parametros `(decimal Amount, DateTime Date, string Description, int AccountId, int CategoryId)`. As regras do projeto (rules.naming.md) recomendam no maximo 3 parametros. Contudo, sendo um record DTO, essa e uma pratica aceitavel e alinhada com os outros DTOs do projeto (ex: `CreateIncomeRequest` segue o mesmo padrao).
- **Sugestao**: Nenhuma acao necessaria -- e um padrao consistente no projeto para DTOs.

## Recommendations

1. **Criar testes para os Update validators** (`UpdateExpenseRequestValidatorTests.cs` e `UpdateExpenseItemRequestValidatorTests.cs`) para garantir cobertura completa dos 4 validators definidos na task.

2. **Alterar `DeleteBehavior.Cascade` para `DeleteBehavior.Restrict`** em `ExpenseItemConfiguration.cs` para alinhar a semantica do banco de dados com a regra de negocio que bloqueia delecao de expenses com itens.

3. **Eliminar a string magica** no `ExpenseRoutes.DeleteAsync` (linha 70) introduzindo um mecanismo de tipo de erro no `Result<T>` ou constantes compartilhadas, evitando acoplamento fragil entre service e controller.

4. **Validar o `CreatedAtAction` cross-controller** manualmente para confirmar que o header `Location` no POST `/api/v1/expenses/{expenseId}/items` retorna a URL correta do item criado.
